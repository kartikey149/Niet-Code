NIET Online Judge — System Blueprint

Full-stack architecture design for a college-exclusive competitive programming platform

React + NestJS Docker Sandbox PostgreSQL + Redis WebSocket Real-time ERP Auth

A production-ready, scalable architecture designed for 5,000+ NIET students — built for performance, security, and developer velocity.

🖥️ Frontend

React 18 + TypeScript

Vite build, Monaco Editor (VS Code engine) for code input, Tailwind CSS, React Query for data fetching

⚙️ Backend API

Node.js + Express / NestJS

REST + WebSocket (Socket.io) for real-time verdicts. JWT auth + ERP integration middleware.

🔒 Judge Service

Python FastAPI + Docker SDK

Isolated microservice. Spawns Docker containers per submission. Enforces CPU/memory/time limits.

🗄️ Database

PostgreSQL + Redis

PostgreSQL for persistent data (users, problems, submissions). Redis for leaderboard (Sorted Sets) & job queues.

📨 Queue

Bull (Redis-backed)

Submission queue to handle traffic spikes gracefully. Workers pull jobs and dispatch to Docker sandbox.

🚀 DevOps

Docker Compose / K8s

Containerized deployment. Nginx reverse proxy. CI/CD with GitHub Actions. Monitoring via Prometheus + Grafana.

💡

Why This Stack?

NestJS provides enterprise-grade structure (decorators, DI, modules) familiar to Spring Boot devs — ideal for a growing student team. FastAPI for the judge is Python-native so Docker SDK integration is seamless. Redis Sorted Sets make real-time leaderboard ranking O(log N) — instant. Monaco Editor gives users VS Code-quality syntax highlighting and autocomplete for free.

The platform must be exclusively accessible to NIET students. Three strategies are provided in order of preference:

🔐

Strategy 1 (Best): ERP OAuth2 / SAML Bridge

RECOMMENDED

Ask NIET's IT department if the ERP supports OAuth2 or SAML. Most modern ERPs (Fedena, EduNxt, MasterSoft) do. Your platform acts as an OAuth2 client.

User clicks 'Login with NIET ERP'
→
Redirect to ERP /authorize
→
User enters ERP creds
→
ERP redirects back with auth code
→
Exchange code for JWT
→
Create/sync local user record
🔐

Strategy 2: LDAP / Active Directory

COMMON

If the college uses Microsoft AD or OpenLDAP (very common), bind to the LDAP server directly using the student's enrollment number + password.

Student submits enrollment + password
→
Backend binds to LDAP server
→
LDAP validates credentials
→
Fetch cn, mail, department attributes
→
Issue platform JWT
🔐

Strategy 3: ERP API Proxy (Fallback)

FALLBACK

If no LDAP/OAuth exists, build a thin proxy that POSTs credentials to the ERP's login endpoint and validates the session response. Requires coordination with IT.

User submits credentials
→
Backend proxies POST to ERP
→
Parse ERP response (200 = valid)
→
Issue platform JWT
→
Cache validity for 1 hour

Docker Sandbox — Security Constraints

// judge-worker/src/runner.ts
import Docker from 'dockerode';

const docker = new Docker();

async function runInSandbox(code: string, language: string, testCase: TestCase) {
  const image = {
    python: 'python:3.11-alpine',
    cpp:    'gcc:12-alpine',
    java:   'eclipse-temurin:17-alpine'
  }[language];

  const container = await docker.createContainer({
    Image: image,
    Cmd: getRunCommand(language),       // e.g. ['python3', 'solution.py']
    AttachStdout: true,
    AttachStderr: true,
    HostConfig: {
      Memory: 256 * 1024 * 1024,        // 256 MB hard limit
      CpuQuota: 50000,                  // 50% of 1 CPU core
      NetworkMode: 'none',              // ❌ No internet access
      ReadonlyRootfs: true,             // ❌ No filesystem writes
      SecurityOpt: ['no-new-privileges'],
      PidsLimit: 64,                    // ❌ No fork bombs
      Ulimits: [{ Name: 'nofile', Soft: 64, Hard: 64 }],
    },
    WorkingDir: '/sandbox',
    Binds: [`${tempDir}:/sandbox:ro`],
  });

  await container.start();

  const result = await Promise.race([
    collectOutput(container),
    timeout(problemTimeLimitMs).then(() => ({ verdict: 'TLE' }))
  ]);

  await container.remove({ force: true });   // Always clean up
  return result;
}

Verdict Decision Logic

function getVerdict(result, expected, timeTaken, memUsed, limits) {
  if (result.compileError)              return 'CE';  // Compile Error
  if (result.runtimeError)             return 'RE';  // Runtime Error  
  if (timeTaken > limits.time_ms)      return 'TLE'; // Time Limit Exceeded
  if (memUsed > limits.memory_mb)      return 'MLE'; // Memory Limit Exceeded
  if (result.stdout.trim() !== expected.trim()) return 'WA'; // Wrong Answer
  return 'AC'; // Accepted ✅
}

Scoring Formula

// Weighted scoring system
const DIFFICULTY_WEIGHTS = { easy: 10, medium: 30, hard: 60 };

// Base score per problem
score = DIFFICULTY_WEIGHTS[difficulty];

// Penalty for wrong attempts before AC
// (inspired by ICPC-style scoring)
penalty = wrong_attempts_before_ac * 5;  // -5 per WA

// Speed bonus: if solved within contest time
speed_bonus = max(0, (time_limit - time_taken_mins) / time_limit * 10);

// Final score added to user's total
final_score = score - penalty + speed_bonus;

Redis Leaderboard Implementation

import { redis } from './redis';

const LEADERBOARD_KEY = 'niet:leaderboard:global';

// Called after every Accepted submission
async function updateLeaderboard(userId: string, scoreToAdd: number) {
  await redis.zincrby(LEADERBOARD_KEY, scoreToAdd, userId);
}

// Get top 50 students (0-indexed)
async function getTopStudents(limit = 50) {
  const entries = await redis.zrevrange(
    LEADERBOARD_KEY, 0, limit - 1, 'WITHSCORES'
  );
  // Hydrate with user details from PostgreSQL
  return hydrateUsers(entries);
}

// Get a student's rank (1-indexed)
async function getUserRank(userId: string) {
  const rank = await redis.zrevrank(LEADERBOARD_KEY, userId);
  return rank !== null ? rank + 1 : null;
}

// Branch-wise leaderboard (e.g. CSE only)
const BRANCH_KEY = (branch: string) => `niet:leaderboard:branch:${branch}`;

// server: emit after leaderboard update

  topChanges: [...],  // Only changed entries

// client: React hook
function useLeaderboard() {
  const [board, setBoard] = useState([]);
  useEffect(() => {
    socket.on('leaderboard:updated', ({ topChanges }) => {
      setBoard(prev => mergeChanges(prev, topChanges));
    });
    return () => socket.off('leaderboard:updated');
  }, []);
  return board;
}

A realistic 6-phase roadmap for a team of 2–4 student developers. Total estimated time: 16–20 weeks.

Phase 1 Foundation & Auth (Weeks 1–2)
1

Setup monorepo

Create /frontend (React + Vite), /backend (NestJS), /judge (FastAPI) with Docker Compose orchestrating all services.

2

Database setup

PostgreSQL with Prisma ORM. Run migrations for users, topics, problems, test_cases, submissions tables.

3

ERP Authentication

Implement LDAP bind or OAuth2 flow. Issue JWTs. Protect all API routes with auth middleware.

4

Basic UI shell

Login page, dashboard layout, nav with topic sidebar. Monaco Editor integration.

Phase 2 Problem Management (Weeks 3–5)
1

Admin panel

CRUD interface for problems — rich markdown editor, difficulty selection, topic assignment, set time/memory limits.

2

Test case manager

Upload bulk test cases (input/output pairs). Mark sample vs hidden. Validate format.

3

Problem viewer

Student-facing problem page: description, constraints, sample I/O, code editor with language selector.

Phase 3 Judge Engine (Weeks 6–9)
1

Docker sandbox

Build the FastAPI judge service. Test Python, C++, Java containers with security constraints from the schema section.

2

Bull queue integration

Connect backend submission endpoint → Redis queue → judge worker. Handle job states: pending/running/done/failed.

3

Verdict pipeline

Implement CE/RE/TLE/MLE/WA/AC logic. Store per-test-case results as JSONB. Update submission record.

4

WebSocket results

Real-time verdict push to the submitting user's browser using Socket.io rooms.

Phase 4 Leaderboard & Profiles (Weeks 10–12)
1

Redis leaderboard

Implement sorted set updates on every AC. Build global + branch-wise boards. Add rank computation.

2

User profile page

Solved problems heatmap (GitHub-style), topic-wise progress bars, submission history, current rank badge.

3

Scoring system

Implement weighted difficulty scoring + WA penalties. Backfill historical submissions.

Phase 5 Polish & Contests (Weeks 13–15)
1

Contest module

Time-bounded contest with its own leaderboard. Freeze standings in last hour (ICPC style).

2

Notifications

Email + in-app notifications for: contest starting, new problems, rank changes.

3

Code plagiarism

Integrate MOSS or custom token-based similarity detection for contest submissions.

Phase 6 Deployment & Hardening (Weeks 16–18)
1

Production Docker Compose

Nginx reverse proxy + SSL (Let's Encrypt). Separate Docker networks for judge isolation.

2

Monitoring

Prometheus metrics + Grafana dashboards for: queue depth, container spawn time, p95 latency.

3

Load testing

Use k6 to simulate 500 concurrent submissions. Tune worker concurrency and container pool.

4

Security audit

Rate limiting, input sanitization, SQL injection prevention (Prisma handles this), OWASP checklist.
